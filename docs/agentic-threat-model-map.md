# Agentic Threat Model Map

This document maps existing lab artifacts to agentic AI threat families without adding runnable exploit payloads. The examples are safe, non-executable, and intended for artifact-reading practice.

| Lab Artifact | Threat Family | What Could Go Wrong | GitHub/MCP Control | Safe Operator Response |
| --- | --- | --- | --- | --- |
| `mcp/broad-server-wildcard-risk.yml` | Excessive agency / overprivileged tool access | A broad wildcard grants more tool access than the agent role needs, including future tools that have not been reviewed. | Explicit MCP tool allowlists, security review, least privilege | Replace broad wildcards with named tools, document the justification, and require review before expanding access. |
| `mcp/local-stdio-risky-example.yml` | Local process execution risk / untrusted tool supply chain | A local tool server could expose unsafe capabilities or run with more local context than the operator intended. | Trusted servers, pinned versions, sandboxing, no unreviewed shell execution | Treat local stdio servers as high risk, review the server source and configuration, and require approval before enabling execution-like tools. |
| `mcp/server-manifest-snapshot.json` | MCP boundary risk / server identity drift | A server name can remain stable while its version, transport, or declared tools change, including dynamic tool-list changes where supported. | Runtime manifest capture, explicit tool review, branch/run provenance | Preserve discovered tool evidence, revalidate wildcard approvals, and require human review for newly discovered capabilities. |
| `artifacts/mcp-boundary-review-report.json` | Tool-output poisoning / confused deputy behavior | An agent may over-trust MCP output or call a tool that acts with broader authority than the agent role should have. | MCP boundary review artifact, strict schemas, provenance checks, human approval for mutating tools | Treat tool output as evidence rather than truth, validate source and schema, and narrow tools to reviewed actions. |
| `.github/agents/executor.agent.md` | Indirect prompt injection / agent instruction tampering | A change to the executor profile could weaken approval requirements or steer the agent toward unsafe execution. | CODEOWNERS, required review for agent profile changes, signed/trusted changes where applicable | Review agent profile diffs like code, verify approval language remains intact, and block execution if instructions drift. |
| `.github/hooks/preToolUse-deny-execute.example.json` | Guardrail tampering / fail-open control risk | A hook decision could be weakened, removed, bypassed, or interpreted as advisory instead of enforcing a block. | Protected hook files, required review, fail-closed policy, audit logging | Keep high-risk hook changes review-gated, verify denied actions remain blocked, and preserve hook decision logs. |
| `artifacts/intent-analysis-payload.json` | Tool execution mediation failure / semantic intent bypass | A model may repackage the same risky capability in a different surface form that raw string matching misses. | Capability-level intent analysis, target/scope classification, pre-tool deny or rewrite | Evaluate requested capability, target resource, scope, and approval state before the tool call reaches execution. |
| `artifacts/hook-approval-binding-report.json` | Approval replay / payload drift | A broad approval can be reused for a different operation or a payload can change after approval. | Payload-bound approval, run/branch/commit binding, expiration window, immediate pre-use recheck | Bind human approval to exact immutable evidence and block execution if the payload changes. |
| `artifacts/hook-enforcement-report.json` | Control-plane bypass / insecure output handling | A blocked hook event may not fail the workflow, or another execution path may bypass the hook layer. | Fail-closed workflow gates, centralized runner policy, protected hook files | Treat blocked hook decisions as blocking workflow status and review alternate execution paths. |
| `memory/stale-memory-drift-scenario.md` | Stale memory poisoning / authorization confusion | Memory can claim the Builder has execution authority even when the current agent profile denies it. | Source-controlled agent profiles, memory validation, human approval gates | Treat memory as untrusted cache and route execution through the authorized Executor profile. |
| `memory/memory-vector-index.lock.json` | Source-of-truth drift / context contamination | Vector memory may be generated from an older commit and then injected into a newer run. | Git-pinned memory binding, index invalidation, pre-context validation | Exclude stale indexes and re-index before privileged execution. |
| `artifacts/memory-authority-review-report.json` | Self-reinforcing false state / control-plane bypass | An agent may rewrite memory or create helper paths to make stale belief look authorized. | Protected memory writes, agent profile review, memory authority review artifacts | Block self-remediation and require human review for authority-affecting memory updates. |
| `artifacts/multi-agent-manifest.json` | Consensus illusion / dropped dissent | Sequential agents may inherit assumptions and produce apparent agreement while dissent or conditional approval remains unresolved. | Multi-agent manifest, independent evidence checks, deterministic status rules | Preserve dissent and conditional approvals; escalate when unresolved findings exist. |
| `artifacts/consolidation-fidelity-report.json` | Semantic compression loss / false approval | A consolidator may downgrade findings, omit conditions, or flatten nuanced risk into a clean final status. | Consolidation fidelity checks, provenance preservation, required human review | Block final approval when risk is omitted, downgraded, or detached from source evidence. |

Threat labels are educational mappings, not authoritative legal or compliance classifications. Exact OWASP naming and numbering may change across versions.
