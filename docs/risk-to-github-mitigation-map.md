# Risk to GitHub Mitigation Map

This document maps common agentic SDLC risks to observable GitHub symptoms and practical GitHub-native mitigations.

| Risk / Anti-Pattern | What It Looks Like in GitHub | Mitigation Using GitHub Controls |
| ------------------- | ---------------------------- | -------------------------------- |
| Planless execution | PR has a diff but no plan, rationale, scope, or success criteria | Require a plan section in the PR template; require review before merge |
| Over-permissioned agents | Workflows can write to the repository or access secrets broadly | Use least-privilege GITHUB_TOKEN permissions; use environments with required reviewers; restrict who can trigger workflows |
| Hidden reasoning | No assumptions, scope, decision trail, or artifact handoff | Require a plan artifact; link workflow runs; record key decisions in PR comments or review artifacts |
| Blind trust in automation | "CI passed, ship it" mindset without human judgment | Combine checks with CODEOWNERS, required reviews, and risk-based approvals |
| Stale memory or context drift | Agent resumes old assumptions that conflict with current repository state | Validate memory against current files, branch, and PR context before execution |
| Memory mistaken for authority | Agent follows stale repository memory or remembered capability instead of current `.agent.md`, workflow, or branch state | Validate memory against current branch, commit SHA, CODEOWNERS-protected agent profiles, workflow permissions, and required review gates before context injection |
| Unreviewed memory mutation | Agent changes long-term memory files or generated memory summaries without review, causing future runs to inherit false state | Protect memory files, require review for memory writes, preserve validation artifacts, and quarantine stale or unpinned memory |
| Consolidated report erases dissent | Final report summarizes upstream security or reviewer concerns as mild warnings or approved status without preserving the original condition | Require artifact schemas that preserve conditional findings, deterministic status merge rules, required reviews, and uploaded evidence artifacts tied to run ID, branch, and commit SHA |
| File existence mistaken for artifact integrity | Workflow checks only that JSON files exist, not whether they are schema-valid, current, provenance-bound, or semantically preserved | Validate schemas, preserve artifact provenance, compare final report fields to upstream risk fields, and fail closed when critical or unresolved findings are omitted |
| Risky MCP expansion | Agent has broad tool access such as `server/*`, newly discovered server tools, or undocumented remote/local MCP boundaries | Use explicit MCP tool allowlists; require MCP boundary review artifacts, CODEOWNERS review, and human approval for new tools, wildcards, remote servers, or local stdio subprocess boundaries |
| Advisory guardrail mistaken for enforced control | Hook logs or comments on risky behavior, but workflow still allows the tool call or records success | Use pre-execution enforcement, protected hook files, required reviews for hook changes, and fail-closed workflow gates |
| Unbound human approval | A broad approval comment or token is reused for a different operation than the one reviewed | Bind approval to run ID, branch, commit SHA, payload hash, environment approval, and expiration window |
| Workflow permission creep | Workflow permissions expand from read-only to write-all without clear need | Review permissions blocks in workflow files; require approval for permission changes |
| Reviewer-note leakage | Internal review comment appears in final published docs or PR output | Add final-render review; require artifact cleanup checks before publish or merge |
| Reviewer note leakage into public artifact | Generated markdown contains internal reviewer notes, unresolved comments, hidden instruction placeholders, or private context references | Use structured output contracts, schema validation, leakage scan artifacts, required reviews, CODEOWNERS for public docs, and fail-closed publication gates |
| Raw model buffer treated as final content | Workflow writes entire model response to docs file instead of extracting approved public payload field | Require `final_output` or `user_payload` extraction, block internal fields, validate promotion readiness, and preserve run ID, branch, and commit evidence |
| Schema-valid artifact treated as decision-ready | Workflow validates required JSON fields and types but does not check whether the content is specific, current, consistent, or safe | Add semantic validation reports, cross-artifact checks, required reviews, provenance-bound artifacts, and fail-closed gates before downstream automation |
| Generic filler passes schema validation | Required fields contain phrases like "looks good," "none," or "make sure it works," while downstream jobs proceed | Flag semantic nulls, disallowed generic phrases, and missing evidence references; require human review when engineering evidence is weak |
| Workflow success mistaken for valid agent handoff | Actions job is green, but downstream agent consumes an empty, stale, partial, wrong-run, or semantically invalid artifact | Use artifact transaction reports, provenance manifests, schema and semantic validation, required reviews, fail-closed workflow gates, and run/branch/commit-bound artifacts |
| Artifact existence mistaken for artifact integrity | Workflow checks only that a file exists before upload or downstream consumption | Check file size, parse status, schema validation, semantic validation, expected record count, content digest, and producing job/run metadata |
| Missing audit evidence | Final report exists but does not link plan, review, logs, or security findings | Require consolidated operator report to list consumed artifacts and unresolved risks |
| Approval bypass | High-risk change merges without human review | Use branch protection, required reviewers, environments, and CODEOWNERS |
