# State Serializability

Multi-agent systems need serializable state contracts because each agent step is usually separated by a workflow boundary, artifact boundary, or tool invocation boundary. The next agent cannot safely rely on conversation memory alone. It needs a deterministic handoff that says what was produced, where it came from, and whether it is valid for the current run.

Workflow outputs are not just strings. In an agentic SDLC pipeline, they are state pointers. A value such as `plan_artifact_name` decides which artifact a downstream Builder Agent consumes. If that value is empty, malformed, stale, or changed mid-run, the downstream agent may continue with the wrong context.

Artifact names must be deterministic. Dynamic names generated by agents can drift because of whitespace, casing, markdown wrappers, punctuation, or illegal characters. For example, `planner-output`, ` Planner-Output `, and `` `planner-output` `` are different handoff strings even if a human reads them as equivalent.

Downstream agents should not infer missing state. If planner state is absent, null, stale, or schema-invalid, the workflow should fail closed before the Builder Agent is invoked.

Stale caches and previous-run artifacts can poison context. A fallback artifact from a prior run may describe a different branch, commit, scope, or risk state. If fallback behavior is allowed, it should be explicitly approved, labeled, and preserved as previous-run context.

| Failure Pattern | Agentic Risk | Safe Control |
| --------------- | ------------ | ------------ |
| empty output string | Builder Agent starts without planner state and guesses a plan | Reject empty outputs and block downstream invocation |
| artifact name with unexpected whitespace | Download searches for a name that was never uploaded | Normalize at the producer and validate exact names |
| artifact name wrapped in markdown backticks | Human-readable formatting changes the machine contract | Reject formatted artifact names in workflow outputs |
| output key renamed mid-run | Downstream job reads a missing key and receives empty state | Version output keys and validate required outputs |
| stale cache fallback | Agent consumes previous-run context as if it were current | Bind state to run ID, commit SHA, and source branch |
| downstream agent processes zero files | Workflow looks successful while agent performed no meaningful work | Treat zero expected files as a blocking silent-success signal |
| final report generated despite missing planner state | Consolidator summarizes an unverified handoff as complete | Require validated planner state before final reporting |
